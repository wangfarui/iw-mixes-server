# Use a durable wardrobe image optimization task module

衣物图片优化采用衣橱域专用的 deep module：MySQL 是任务状态与持久队列的唯一权威来源，Redis 状态副本被移除。任务由衣物 ID、不可变衣物源图 URL、规范化提示词和优化规则版本确定稳定身份；失败后的主动重试保留任务 ID，并创建递增且可隔离迟到结果的任务尝试。

## Decisions

- 同一衣物最多有一个 `queued` 或 `running` 任务；相同指纹的启动请求幂等返回现有任务，不同指纹的请求被拒绝。
- 不同衣物允许并行，生产初始并发数为 1；正确性不依赖单线程或单实例。
- `queued` 任务必须在 `iw-core` 重启后恢复；`running` 任务尝试默认有可配置的 15 分钟期限，超时后失败且不自动重试。
- 已持久化外部任务 ID 的尝试在重启后继续查询；无法恢复的同步供应商调用失败并等待用户主动重试。
- 任务成功只发生在生成结果已上传并成功关联为衣物优化图之后。
- 任务活跃期间禁止修改衣物源图，普通元数据修改不影响已持久化的输入快照；衣物删除会把任务置为不可重试的 `cancelled`。
- 用户主动删除衣物优化图不会改写成功的任务尝试；相同输入再次发起时，在原任务下创建新尝试。
- 文件引用和任务成功在同一个数据库事务中提交，OSS 删除在提交后执行。待删除文件先持久化，按有限退避策略重试，最终失败保留为 `manual_required` 供人工处理。
- 任务、任务尝试和文件清理记录首版长期保留，仅随用户注销的数据清理规则删除。
- 衣橱只保留专用任务、尝试和清理模型；共享能力止于 `iw-external` 的图片生成 seam 与 provider adapters。第二种图片业务出现前不建立通用任务表、父类或 `businessType` 分支。

## Interface

页面可启动任务、重试失败任务、按任务 ID 查询、查询衣物当前任务及删除当前衣物优化图。衣物 module 可在删除衣物时取消活跃任务，并在修改衣物源图前校验是否允许；任务认领、恢复、期限、provider 状态归一化、文件落位与清理重试均隐藏在 implementation 内。

启动 interface 只接收衣物 ID 与可选用户提示词。task module 从已保存的衣物数据构造输入快照和指纹；未保存的页面修改必须先保存。

## Rejected alternatives

- Redis 任务投影：拒绝，因为它复制权威状态并引入双写、TTL 与恢复分支。
- 同一衣物以“最新完成者获胜”并行生成：拒绝，因为旧结果可能覆盖用户当前意图。
- 通用 `ai_image_generate_record` 模型：拒绝，因为当前只有一个业务 adapter，通用 seam 尚属 hypothetical。
- 在数据库事务内同步删除 OSS 文件：拒绝，因为删除成功而事务回滚会产生悬空引用；允许孤儿文件并持久重试更安全。
- 首版开放手动取消：拒绝，因为多数 provider adapter 无法撤销已发出的调用，且会扩大 interface 与 UI 状态。
