# 衣物图片优化任务持久化重构研发规格

> 后续变更：参考图 provider 契约、任务尝试中的外部任务字段、同步调用租约和 provider 验收方式，以 [AI 参考图 Provider Module 重构研发规格](./reference-image-provider-module.md) 与 ADR 0002 为准。本文件保留第一阶段任务 module 的原始设计背景。

## Problem Statement

衣橱用户当前可以为已保存的衣物发起图片优化，但任务生命周期分散在数据库记录、Redis 临时状态、进程内线程池、远程图片生成调用和 OSS 文件操作之间。用户离开页面或 `iw-core` 重启后，任务状态和执行进度可能无法可靠恢复；同一衣物也缺少清晰、持久且可并发校验的任务边界。

当前实现还把衣橱业务存入通用的 AI 图片生成记录。该模型混合了业务任务身份、供应商执行状态、去重、结果文件和缓存恢复语义，无法准确表达“一个衣物图片优化任务可以有多次任务尝试”“删除衣物优化图不等于历史尝试失败”“衣物删除后迟到结果只能清理”等衣橱领域规则。

此外，文件引用变更与 OSS 删除目前缺少持久化清理机制。同步删除失败后没有可靠重试和人工处理线索，数据库事务回滚与外部文件删除之间也存在产生悬空引用的风险。

## Solution

为衣橱域建立专用、持久化的衣物图片优化任务模块。MySQL 作为任务状态、持久队列和恢复信息的唯一权威来源；任务模块统一负责身份判定、输入快照、排队、执行、重试、期限、迟到结果隔离、衣物生命周期联动、衣物优化图落位以及文件清理记录。

用户仍在衣物编辑页通过一个状态驱动的主按钮完成操作：无任务时“一键优化”，任务执行中显示“优化中…”，失败后显示“重新优化”，已有当前衣物优化图时显示“已优化”。用户删除衣物优化图后可以再次发起优化；历史成功尝试仍被保留，OSS 文件删除通过持久化清理机制可靠完成。

远程图片生成能力继续由 `iw-external` 提供通用 provider adapter，但衣橱任务生命周期不抽象成跨业务父模型。未来出现第二种图片生成业务时，先建立该业务自己的领域模块，再从两个真实 adapter 中提取已经被证明的共性。

## User Stories

1. 作为衣橱用户，我希望只凭已保存的衣物和可选补充要求发起图片优化，从而无需由页面拼装或维护服务端优化规则。
2. 作为衣橱用户，我希望未保存衣物不能发起优化，从而确保任务始终关联到稳定的衣物身份。
3. 作为衣橱用户，我希望任务使用已保存的衣物源图，从而避免未保存页面状态与后台任务输入不一致。
4. 作为衣橱用户，我希望相同衣物、相同输入的重复点击返回同一活跃任务，从而避免重复扣费和重复生成。
5. 作为衣橱用户，我希望同一衣物已有活跃任务时不能再启动不同输入的新任务，从而避免多个结果竞争覆盖。
6. 作为衣橱用户，我希望不同衣物的优化任务互不阻塞业务规则，从而让系统未来可以安全提高并发能力。
7. 作为衣橱用户，我希望离开并重新进入衣物编辑页后仍能看到当前任务，从而不必一直停留在等待页面。
8. 作为衣橱用户，我希望 `iw-core` 重启后排队中的任务可以继续执行，从而不因服务重启丢失请求。
9. 作为衣橱用户，我希望可恢复的远程任务在服务重启后继续查询原任务，从而避免重复调用供应商。
10. 作为衣橱用户，我希望无法恢复的供应商调用明确失败并等待我的主动重试，从而不会在未知情况下产生第二次计费调用。
11. 作为衣橱用户，我希望运行过久的任务在明确期限后失败，从而不会永久显示“优化中”。
12. 作为衣橱用户，我希望失败任务可以通过“重新优化”主动重试，从而无需创建一个语义不明的新任务。
13. 作为衣橱用户，我希望重试沿用原任务及其输入快照，从而可以查看一次业务意图下的完整尝试历史。
14. 作为衣橱用户，我希望改变优化输入后发起的是新任务，从而让不同优化意图彼此可区分。
15. 作为衣橱用户，我希望旧任务尝试的迟到结果不会覆盖新尝试结果，从而始终看到当前有效结果。
16. 作为衣橱用户，我希望只有图片已上传、落位并关联到衣物后任务才显示成功，从而不会看到无法使用的“成功”状态。
17. 作为衣橱用户，我希望执行期间仍可修改衣物名称、分类、标签、品牌、材质和备注等普通元数据，从而不被长耗时任务阻断日常编辑。
18. 作为衣橱用户，我希望普通元数据修改不改变正在执行任务的输入快照，从而保证任务结果可解释和可追踪。
19. 作为衣橱用户，我希望任务排队或运行期间不能更换衣物源图，从而避免生成结果关联到错误的源图。
20. 作为衣橱用户，我希望衣物删除不被后台任务阻止，从而可以正常管理衣橱数据。
21. 作为衣橱用户，我希望衣物删除后活跃任务变为不可重试的已取消状态，从而不会继续产生可安装结果。
22. 作为衣橱用户，我希望衣物删除后的迟到生成文件不会重新关联到已删除衣物，从而避免数据复活和孤儿业务引用。
23. 作为衣橱用户，我希望可以主动删除当前衣物优化图，从而恢复使用衣物源图作为展示图。
24. 作为衣橱用户，我希望删除衣物优化图后，搭配和穿着记录中的对应图片引用回退到衣物源图，从而避免继续展示待删除文件。
25. 作为衣橱用户，我希望主动删除衣物优化图不会把历史成功尝试改写成失败，从而保留真实的任务历史。
26. 作为衣橱用户，我希望删除当前衣物优化图后可以用相同输入再次优化，从而获得新的结果。
27. 作为衣橱用户，我希望页面只出现一个具有清晰状态语义的优化主按钮，从而不会在“一键优化”和“重新优化”之间困惑。
28. 作为衣橱用户，我希望排队和运行时按钮不可重复点击并显示进度状态，从而明确知道系统已接受请求。
29. 作为衣橱用户，我希望失败时看到可理解的错误提示和可重试状态，从而知道下一步可以做什么。
30. 作为衣橱用户，我希望已取消任务不显示可重试操作，从而不会对已删除衣物发起无效调用。
31. 作为衣橱用户，我希望只能查询和操作属于自己的衣物图片优化任务，从而保护个人数据和生成结果。
32. 作为系统维护者，我希望 MySQL 是任务状态的唯一权威来源，从而无需排查 Redis 与数据库不一致。
33. 作为系统维护者，我希望任务队列能够从持久化数据自动恢复，从而不依赖单个 JVM 的内存状态。
34. 作为系统维护者，我希望任务认领在多实例下仍然互斥，从而不会重复执行同一任务尝试。
35. 作为系统维护者，我希望生产初始执行并发数可配置为 1，从而在控制供应商压力的同时保留未来扩容能力。
36. 作为系统维护者，我希望任务尝试记录供应商任务身份、期限、错误和结果，从而能够诊断每次实际执行。
37. 作为系统维护者，我希望旧文件删除在数据库事务提交后执行，从而不会因事务回滚产生悬空引用。
38. 作为系统维护者，我希望每个待删除 OSS 文件都有持久化记录和有限退避重试，从而让瞬时删除失败可以自动恢复。
39. 作为系统维护者，我希望多次清理失败的记录进入 `manual_required` 并保留上下文，从而可以后期人工清理。
40. 作为系统维护者，我希望上传成功但数据库落位失败的新文件也进入相同清理机制，从而不会形成不可追踪的 OSS 孤儿文件。
41. 作为系统维护者，我希望任务、任务尝试和清理记录首版长期保留，从而具备审计与故障追踪依据。
42. 作为系统维护者，我希望用户注销时这些记录遵循用户数据清理规则，从而不额外保留已注销用户的数据。
43. 作为后续业务开发者，我希望通用边界只覆盖远程图片生成能力，从而可以复用 provider adapter 而不继承衣橱特有生命周期。
44. 作为后续业务开发者，我希望在第二种图片业务出现前不引入通用父表、父类或 `businessType` 分支，从而避免被假设性的抽象束缚。

## Implementation Decisions

- 影响范围为后端衣橱域和原生微信小程序衣物编辑页。Web 管理端不需要修改。
- 在衣橱域建立专用的衣物图片优化任务 deep module。该模块封装任务身份、状态机、任务尝试、持久队列、任务认领、恢复、期限、远程执行协调、结果落位和清理意图；页面、衣物服务和调度设施只依赖其公开行为。
- 移除图片优化任务的 Redis 状态和进程内状态权威。Redis 不保留任务投影、TTL 或恢复分支；MySQL 是查询、并发校验、排队和重启恢复的唯一权威来源。
- 以“衣物 ID、不可变衣物源图 URL、最终规范化提示词、服务端优化规则版本”计算稳定任务指纹。相同指纹代表同一个衣物图片优化任务，用户 ID 另行用于数据归属和权限校验。
- 衣物源图 URL 可以作为源图身份，是因为当前文件上传使用不可复用的对象地址，且业务已确认 URL 对应内容不会被替换。
- 优化规则版本是服务端拥有的稳定标识，例如 `wardrobe-item-image-optimize/v1`。它描述输出语义规则，不暴露模型名称；仅在提示词规则、输出目标或需要有意失效旧身份时升级，不因重试、日志、存储键或调度实现变化而升级。
- 启动操作只接收衣物 ID 和可选用户提示词。服务端读取已保存的衣物源图及元数据，生成规范化输入快照和任务指纹。页面不得提交源图 URL、规则版本或完整服务端提示词。
- 未保存的表单修改必须先保存再启动任务。普通元数据在任务启动时进入不可变输入快照；任务活跃期间的后续普通元数据修改不影响当前任务执行、生成和结果保存。
- 同一衣物最多存在一个状态为 `queued` 或 `running` 的活跃任务。相同指纹的启动请求幂等返回该任务；不同指纹的启动请求返回业务冲突及当前任务信息。
- 任务状态为 `queued`、`running`、`succeeded`、`failed`、`cancelled`。`cancelled` 首版只由衣物删除触发，不提供用户手动取消接口。
- 失败后的用户主动重试保持任务 ID 不变，创建递增编号的新任务尝试，并把任务重新置为 `queued`。重试复用原任务输入快照；若当前衣物源图已与任务快照不一致，则拒绝重试并要求按当前输入启动新任务。
- 每个任务尝试持有独立尝试编号或等价 fence token。写入外部任务 ID、状态、结果和衣物优化图时必须同时校验当前尝试身份；旧尝试或超时尝试的迟到结果不得覆盖当前尝试。
- 衣物图片优化任务表至少持久化：任务 ID、用户 ID、衣物 ID、任务指纹、衣物源图 URL、用户提示词、规范化提示词、优化规则版本、输入元数据快照、任务状态、当前尝试编号、当前衣物优化图信息、结果删除时间、错误摘要以及创建、更新和完成时间。
- 任务尝试表至少持久化：任务 ID、尝试编号、状态、执行节点或认领标识、开始时间、期限、供应商类型、外部任务 ID、供应商返回信息、生成结果信息、失败原因以及创建、更新和完成时间。任务 ID 与尝试编号组合必须唯一。
- 同一任务指纹在用户和衣物范围内唯一；同一衣物活跃任务的唯一性必须由数据库事务、锁或等价条件写保证，不能依赖前端禁用按钮、单线程或单实例。
- 调度器从 MySQL 原子认领 `queued` 尝试。不同衣物可以并发；生产初始并发配置为 1。实现必须允许以后仅通过配置提高并发数，并保证多实例不会重复认领。
- `queued` 任务在应用启动和周期扫描时自动恢复。`running` 尝试默认期限为可配置的 15 分钟；超过期限后转为 `failed`，不自动重试。
- 若任务尝试已持久化可查询的外部任务 ID，重启后继续查询原供应商任务，但仍受原 15 分钟期限约束。若供应商是同步调用或没有可恢复身份，进程中断后不得自动重发供应商请求；该尝试失败并等待用户主动重试。
- 远程图片生成能力继续通过 `iw-external` 的通用接口和 provider adapters 提供。衣橱模块负责解释领域输入与输出，不把任务、尝试、衣物删除或文件清理语义下沉到通用远程服务。
- 任务只有在供应商结果已成功上传，并在数据库事务中通过当前尝试校验、衣物存在校验和业务引用替换后，才可以标记为 `succeeded`。
- 结果落位顺序为：上传新文件；开启数据库事务并校验任务尝试；写入当前衣物优化图引用；替换搭配、穿着记录和封面中的旧优化图引用；标记尝试与任务成功；为被替换旧文件写入清理意图；提交事务；提交后异步删除 OSS 文件。
- 若新文件上传成功但数据库事务未提交，应为该新文件建立持久化清理记录。系统优先容忍可追踪的孤儿文件，不允许产生指向已删除文件的数据库引用。
- 任务活跃期间，衣物更新操作必须比较衣物源图并拒绝源图变化；其他元数据字段可以正常修改。该约束必须由后端执行，微信小程序的禁用状态只用于改善体验。
- 删除衣物时，在同一业务事务中把活跃任务标记为不可重试的 `cancelled`。排队任务不再执行；运行任务的迟到结果不落位，已生成或已上传文件进入持久化清理机制。
- 主动删除衣物优化图时，在数据库事务中立即移除当前优化图业务引用，并把搭配、穿着记录与相关封面引用回退到衣物源图，同时写入 OSS 清理意图。接口不等待 OSS 删除完成。
- 主动删除衣物优化图不得把已经成功的任务尝试改成失败。任务保留历史结果 URL、完成时间和结果删除时间；相同输入再次启动时，在原任务下创建新尝试并重新排队。
- 文件清理采用衣橱任务内部专用持久模型，首版不抽取到全局文件清理模块。清理记录至少包含文件 URL、用户 ID、衣物 ID、任务 ID、任务尝试、清理原因、状态、重试次数、下次执行时间、最后错误和最后尝试时间。
- 文件清理状态至少包括待处理、重试中、已完成和 `manual_required`。自动重试间隔依次为 1 分钟、5 分钟、30 分钟、2 小时、12 小时；最后一次仍失败时停止自动重试，保留记录供人工处理。
- 清理原因至少区分：替换旧衣物优化图、用户主动删除衣物优化图、衣物删除后的迟到结果、上传后数据库事务失败产生的孤儿文件。清理操作必须幂等，OSS 文件已不存在视为成功。
- 任务、所有任务尝试和文件清理记录首版不设置自动 TTL。`manual_required` 记录在人工确认处理前持续保留；用户注销时遵循统一用户数据清理策略。
- 现有通用 AI 图片生成记录迁移为衣橱专用任务和任务尝试。迁移必须保留可识别的任务身份、输入、供应商任务 ID、状态、结果 URL、错误和时间信息；已有成功结果及业务文件引用不得丢失。无法可靠恢复的历史处理中记录应归一为失败，除非其外部任务 ID 仍可查询。
- 数据迁移完成并校验衣橱不再读写旧模型后，移除旧通用记录在衣橱域中的实体、DAO 和业务类型分支；当前没有第二种业务需要保留该通用表。
- 页面侧保留一个状态驱动的优化主按钮：不存在任务且没有当前结果时显示“一键优化”；`queued` 或 `running` 时显示“优化中…”并禁用；`failed` 时显示“重新优化”；当前结果有效且任务成功时显示“已优化”并禁用。删除衣物优化图继续使用独立的“删除”操作。
- 页面进入或恢复前台时按衣物 ID 查询当前任务；执行中按任务 ID 轮询。页面轮询超时只结束本地等待提示，不得擅自把服务端任务标记失败；重新进入页面后以服务端状态恢复。
- 页面在任务活跃期间禁用衣物源图更换入口，但普通表单字段保持可编辑。后端冲突或源图锁定错误需显示明确提示。
- 保留现有启动、按任务 ID 查询、按衣物查询当前任务和删除衣物优化图的高层接口形状，新增按任务 ID 重试失败任务的独立操作。查询响应至少返回任务 ID、衣物 ID、标准状态、当前尝试编号、当前衣物优化图、错误摘要及是否可重试。
- 所有启动、查询、重试、删除和衣物联动操作都必须验证当前用户对衣物与任务的所有权，不能仅按公开任务 ID 访问。
- 任务认领、恢复、期限扫描、provider 状态归一化、文件上传、结果落位和清理重试均为模块内部实现，不暴露为页面 API。

## Testing Decisions

- 测试只断言公开行为和持久化后的可观察结果，不直接测试 DAO 调用顺序、线程池实现、调度器私有方法、Redis 键、表结构细节或私有状态转换函数。
- 首选的最高测试 seam 是衣物图片优化任务模块的应用级公开接口：启动、重试、按任务查询、按衣物查询、删除当前衣物优化图，以及衣物模块的删除与源图修改行为。通过可控时钟、可控远程图片生成 adapter 和可控文件存储 adapter 驱动异步场景。
- 后端应用级测试应覆盖：首次启动创建排队任务；相同指纹幂等；同衣物不同指纹冲突；不同衣物可独立认领；启动参数只依赖衣物 ID 与可选提示词；越权访问被拒绝。
- 后端状态测试应覆盖：`queued` 到 `running` 到 `succeeded`；供应商失败；15 分钟期限失败；用户主动重试创建新尝试；旧尝试迟到结果被 fence；`cancelled` 不可重试。
- 后端恢复测试应使用同一测试数据库重建应用级模块实例，验证排队任务在重启后被认领、带外部任务 ID 的运行任务继续查询、无可恢复身份的运行调用不会被重新发送。
- 后端衣物联动测试应覆盖：普通元数据更新成功且不改变任务快照；活跃期间源图更新被拒绝；衣物删除取消任务；衣物删除后的迟到结果不落位并产生清理记录。
- 后端文件一致性测试应覆盖：成功结果落位后才标记任务成功；替换结果时业务引用原子切换；数据库提交失败时新文件进入清理；主动删除结果时引用立即回退且历史尝试保持成功；OSS 删除失败按既定退避重试并最终进入 `manual_required`。
- 后端迁移测试应以代表性的处理中、成功、失败和已有衣物优化图记录验证数据映射，确保现有成功图片引用和历史信息不丢失。
- HTTP 合同测试应覆盖标准状态值、重试资格、业务冲突、任务不存在、衣物不存在、所有权校验和错误信息，不要求测试 Controller 内部委托细节。
- `iw-external` 只补充或延续 provider adapter 的合同测试：同步结果、异步外部任务 ID、查询成功、查询失败和响应归一化。现有 AI 服务的 provider URL 与请求组装单元测试可作为该层 prior art。
- 衣橱域当前没有现成任务测试，因此新增测试应参考仓库中服务级测试使用 mock adapter 和可控依赖的方式，同时提高到衣物图片优化任务公开行为 seam，避免复制当前大型服务实现的内部结构。
- 微信小程序验证应覆盖按钮文案与禁用状态映射、进入页面恢复任务、轮询成功与失败、显式重试、删除优化图回退源图、活跃期间不能更换源图、普通字段仍可编辑，以及后端冲突提示。
- 微信小程序当前没有自动化测试框架，本需求不为单页引入重量级框架。最低验证包括路由检查、目标脚本语法检查和微信开发者工具编译；关键状态流通过可控接口响应或模拟环境进行页面验证。
- 端到端验收至少覆盖一次真实本地链路：保存衣物、启动优化、离开并返回页面、观察恢复、生成成功、删除衣物优化图、确认展示回退和清理记录完成。供应商不可用时可以使用遵循同一合同的测试 adapter。

## Out of Scope

- 首版不提供用户手动取消衣物图片优化任务；`cancelled` 仅由衣物删除触发。
- 首版不自动重试失败的图片生成任务，所有生成重试均由用户主动发起。
- 首版不建立跨业务通用任务表、通用任务父类、`businessType` 分支或全局图片生成工作流引擎。
- 首版不把衣橱文件清理记录抽取为 `iw-web` 或其他公共模块的通用清理中心。
- 首版不提供文件清理管理页面；`manual_required` 通过持久化记录、日志和数据库查询支持人工处理。
- 首版不设置任务、尝试和清理记录的自动归档或 TTL。
- 不允许任务执行期间更换衣物源图后继续沿用原任务；源图变化必须等待活跃任务结束并启动新任务。
- 不因普通衣物元数据修改取消、失败或阻止当前任务结果落位。
- 不要求供应商具备真正的远程取消能力，也不因本地任务取消而假设供应商调用可撤销。
- 不修改 Web 管理端，也不新增管理端任务列表或监控页面。
- 不调整“识别当前图片”等其他衣橱 AI 能力的业务语义。

## Further Notes

- 本规格使用领域术语“衣物源图”“衣物优化图”“衣物图片优化任务”和“任务尝试”。实现和接口文案应避免重新使用含义模糊的“AI 图片记录”“临时任务”或把任务重试称为“新任务”。
- 本规格遵循已接受的架构决策“Use a durable wardrobe image optimization task module”。如实现过程中需要重新引入 Redis 状态、通用任务模型、同步事务内 OSS 删除或公开手动取消，应先更新 ADR 并重新确认取舍。
- 当前代码将任务生命周期集中在大型衣橱助手服务中，并通过 Redis、静态单线程执行器和通用图片生成记录协调。实施时应把生命周期收拢到专用模块，但不应顺带重构衣橱搭配建议或图片识别功能。
- 当前生产仅有一个 `iw-core` 进程，但数据模型、认领和 fence 规则必须按多实例正确性设计，避免未来扩容时重新改造核心状态机。
- OSS 清理是最终一致过程。面向用户的删除成功表示业务引用已删除并已登记可靠清理意图，不表示远程对象一定在响应返回前物理删除。
